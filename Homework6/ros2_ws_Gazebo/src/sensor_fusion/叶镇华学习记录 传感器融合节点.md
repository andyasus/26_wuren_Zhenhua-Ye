# sensor_fusion —— 传感器融合节点
## 此部分由25级机器人工程叶镇华完成

## 我做了什么

写了一个 ROS2 节点，把 GPS、IMU、磁力计、里程计四个传感器的数据用 EKF 融合起来，输出车辆位姿。

代码结构：
- `sensor_fusion_node.py` — ROS2 节点，订阅/发布，定时器驱动
- `ekf_localization.py` — 5 维 EKF（px, py, yaw, v, yaw_rate），运动模型预测 + 各传感器观测更新
- `gps_converter.py` — 经纬度转米制坐标，第一帧定位原点

订阅的话题：
| 话题 | 类型 | 用途 |
|------|------|------|
| `/gps/fix` | NavSatFix | 位置观测 |
| `/imu/data` | Imu | 航向角速度观测 |
| `/mag/data` | MagneticField | 航向角观测 |
| `/odometry` | Odometry | 速度观测 |

发布的话题：
| 话题 | 类型 | 用途 |
|------|------|------|
| `/vehicle_pose` | PoseStamped | 融合后的车辆位姿 |

## 编译

```bash
cd ros2_ws_Gazebo
source /opt/ros/humble/setup.bash
colcon build --packages-select sensor_fusion
source install/setup.bash
```

## 运行

单独启动融合节点：
```bash
ros2 launch sensor_fusion fusion_launch.launch.py
```

跟仿真一起启动：
```bash
ros2 launch car_sim sim_bringup_harmonic.launch.py with_fusion:=true
```

## 学习记录

- **EKF 原理**：之前只写过简单卡尔曼滤波，这次把完整的预测-更新流程走了一遍。运动模型被速度 v 和角速度 yaw_rate 驱动，状态转移矩阵 F 手动推导雅可比。
- **GPS 转局部坐标**：把经纬度差值投影到切平面，用地球半径算米制偏移。第一帧设为原点 (0, -15)，方便跟赛道坐标对齐。
- **ROS2 多话题订阅**：之前用的少，这次一个节点同时订阅四个话题，用回调函数分别更新 EKF 的不同观测维度，定时器统一发布。
- **Gazebo + 桥接**：第一次接触 Gazebo Harmonic 和 ros_gz_bridge 的配合，理解了 protobuf 版本不兼容的问题。
- **坑**：ros_gz_bridge 的 apt 版链接的是 Ignition Fortress 的库，跟 Harmonic 的 gz-transport13 不兼容，需要用 Python API 或重新编译桥接才能通。
- **AI使用情况说明**：用的不少，但代码都是我一行一行手敲的。至于为什么用的不少，那就得归到我一坨的线性代数基础头上了。卡尔曼滤波的算法用到了很多矩阵运算，在AI的帮助下我理解了基本原理，然后让AI提供一点思路引导我手敲代码。还有用来修正launch文件的一些小错误。**最后的联调已经来不及了，很遗憾超时提交作业。6月20日由我使用DeepSeek全面修复了项目文件夹的所有错误，补交大作业，这个大作业终于告一段落。**
